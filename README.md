# ArenaFlow Nexus

**A living digital twin of the stadium that predicts problems before they happen.**

An AI-powered operations platform for FIFA World Cup 2026 host stadiums. The primary UI is not a chatbot but a real-time **operations brain** — a network of live stadium nodes (gates, food courts, washrooms, medical stations, exits, parking) whose state is continuously read by a generative AI layer that produces predictive operational alerts.

A floating multilingual AI assistant sits above the dashboard for fans and staff to ask on-demand questions grounded in the same live state.

## Feature map — Challenge tracks

| Challenge track | Implementation |
| --- | --- |
| **Dynamic crowd management** | `src/lib/stadium.ts` simulation engine; `NodeCard` with live density/wait/noise sparklines; halftime and pre-kickoff surge modeling calibrated to the real match schedule. |
| **Smart indoor navigation** | `/navigation` route: stylized SVG floor plan + ranked least-congested route to food/washroom/exit/medical based on live density. |
| **Real-time decision support (Predictive AI)** | `PredictionsFeed` calls `predictOps` server function every 30s. Gemini-3-flash (via Lovable AI Gateway) receives full JSON node snapshot + weather + match context and returns structured `{ alerts, summary }` JSON. Rendered center-right on the dashboard. |
| **Multi-language assistance** | i18n (en/hi/es/fr/ar) with RTL for Arabic; language passed to every AI prompt so alerts and assistant replies respond in the selected language. |

## Real data sources

- **Weather**: [Open-Meteo](https://open-meteo.com/) (no key). Fed into the simulation engine — hot days push medical/washroom pressure up.
- **Live match data**: [TheSportsDB](https://www.thesportsdb.com/) free tier. Real fixtures drive real kickoff times; the simulator applies pre-kickoff and halftime surges based on the actual event schedule.
- **Simulated sensor layer**: crowd density, wait times, noise, staff availability, parking occupancy. There is no public live sensor feed for 2026 stadiums, so these are generated by a bounded random-walk engine **calibrated to real match schedule and weather data**. See `src/lib/stadium.ts` for the model.

## AI layer

Lovable AI Gateway → `google/gemini-3-flash-preview`. Two entry points:

- `predictOps` — receives a compact node-state snapshot on a 30s cadence, returns strict JSON `{ summary, alerts: [{ severity, node, prediction, recommendedAction, etaMinutes }] }`.
- `askAssistant` — grounded Q&A. Every question is answered against the current live node snapshot.

Both run server-side (`src/lib/ai.functions.ts`). The `LOVABLE_API_KEY` is never exposed to the browser.

## Team theming & notifications

- `src/lib/teams.ts` — 24 national teams with official primary/accent hex.
- `ThemeSync` writes `--team-primary`, `--team-accent`, `--team-gradient`, and a contrast token to `<html>` so the FAB, gradients, and highlight rings repaint reactively.
- The user's selected team is persisted (Zustand + `persist`).
- Live in-app toasts (and browser push, when permitted) fire when the user's team has a match kicking off within 60 minutes.

## Non-negotiables

- **Security**: `LOVABLE_API_KEY` lives on the server only. All AI calls run through `createServerFn`.
- **A11y**: dark base with high-contrast overlays; keyboard-reachable FAB; `aria-label` / `aria-pressed` throughout; `dir="rtl"` when Arabic is active.
- **Tests**: `src/lib/__tests__/stadium.spec.ts` — pure-function tests for the simulation engine (density bounds, halftime pressure, severity escalation). Run with `bunx vitest run`.
- **Type-safety**: strict TS; every node metric is typed via `StadiumNode`.
